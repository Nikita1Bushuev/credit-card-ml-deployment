# Credit Card ML Deployment

Сервис прогнозирования дефолта по кредитным картам на основе датасета `Default of Credit Card Clients Dataset`. Реализован как Flask REST API, упакован в Docker-контейнер.

## Описание проекта
**Цель проекта**: разработать и внедрить в production-like-среду сервис машинного обучения для прогнозирования дефолта по кредитным картам, который охватывает полный цикл от сохранения модели до организации A/B-тестирования.

- **Модель**: RandomForestClassifier
- **Фреймворк**: Flask
- **Контейнеризация**: Docker + Docker Compose
- **Версионирование**: Git, Docker Hub

## Инструкция по запуску (локально и в Docker)

### Запуск локально
1. Клонировать репозиторий и перейти в папку:
    ```powershell
   git clone [<repo-url>](https://github.com/Nikita1Bushuev/credit-card-ml-deployment.git)
   cd credit-card-ml-deployment

2. Создать и активировать виртуальное окружение:
    python -m venv venv
    .\venv\Scripts\Activate.ps1 

3. Установить зависимости:
    pip install -r requirements.txt

4. Запустить Flask-сервис:
    python -m app.api

Сервис будет доступен на http://localhost:5000

### Запуск через Docker
1. Собрать образ:
    docker build -t credit-card-ml-service:v1 -f docker/Dockerfile .

2. Запустить контейнер:
    docker run -d -p 5000:5000 --name ml-service credit-card-ml-service:v1

3. Проверить работоспособность:
    curl http://localhost:5000/health

### Запуск через Docker Compose
docker-compose up -d
Остановка: docker-compose down.

## Описание формата запросов и ответов
**POST /predict**

Запрос:
{
  "features": [массив из 23 чисел]
}

features - упорядоченный список признаков клиента в том же порядке, что и столбцы обучающего датасета (исключая default.payment.next.month)

Ответ:
{
  "model_version": "v1",
  "prediction": 1,
  "probability": 0.6854
}

prediction - 1 (дефолт ожидается) или 0.
probability - вероятность класса 1 (дефолт).
model_version - версия используемой модели.

**GET /health**
Ответ:
{"status": "healthy"}

## Архитектура сервиса (монолит vs микросервисы)
В данном проекте выбран **монолитный подход** (один Flask-сервис, содержащий всю логику).
Данный выбор был сделан так как:
- Модель одна, инференс происходит синхронно, запросы однотипны
- Нагрузка не требует разделения на независимые сервисы (микросервисы добавили бы излишнюю сложность - сетевые задержки, управление контрактами).

## Концепты MLOps


### Брокер сообщений (RabbitMQ)
При росте нагрузки и необходимости асинхронной обработки можно использовать RabbitMQ. Клиент отправляет запрос в очередь, worker забирает пачку сообщений, выполняет батч-предсказание и возвращает результат. Это увеличивает пропускную способность, позволяет балансировать нагрузку и централизованно логировать запросы.

### Логирование и мониторинг
Логи запросов и ответов лучше писать в формате JSON. В production-системе логи собираются через Filebeat в Elasticsearch, индексируются и визуализируются в Kibana (ELK-стек). Такой подход обеспечивает наблюдаемость сервиса и быстрое выявление аномалий.

### DVC (Data Version Control)
DVC - система контроля версий, которая позволяет фиксировать состояние датасета, воспроизводить эксперименты и легко переключаться между версиями данных. В жизненном цикле модели DVC полезен для отслеживания изменений в обучающей выборке и синхронизации с кодом модели.

### MLflow
MLflow - платформа для управления полным жизненным циклом моделей машинного обучения, отслеживает эксперемнты: запись параметров, метрик, артефактов (моделей). Используется для сравнения разных версий модели, регистрации лучшей. В проекте мог бы применяться для логирования метрик и экспорта моделей.

### ONNX-ML
ONNX-ML предназначен для упаковки и запуска моделей традиционного машинного обучения. Оно позволяет сохранять алгоритмы из библиотек (например, деревья решений, линейную регрессию) для их дальнейшего использования на любых платформах и языках. Модели scikit-learn можно конвертировать в формат ONNX/ONNX-ML с помощью `skl2onnx`. Это обеспечивает переносимость между платформами и языками программирования, а при использовании ONNX Runtime может также повысить производительность инференса за счёт оптимизаций выполнения.

### uWSGI + NGINX
В production Flask-приложение запускают через WSGI-сервер (uWSGI или Gunicorn), который управляет воркерами и обеспечивает надёжную обработку запросов. Перед ним размещают NGINX в роли обратного прокси для приёма входящих соединений, SSL/TLS-терминации, отдачи статических файлов и, при необходимости, балансировки нагрузки между экземплярами приложения.

## Бизнес-метрики

**Снижение ожидаемых финансовых потерь**
Рассчитывается как разница между ожидаемыми потерями без модели (например, одобрение всех заявок) и с моделью. Формула:  
   `Expected Loss = ∑ (PD_i * LGD_i * EAD_i)`,  
   где PD — вероятность дефолта (выход модели), LGD — доля потерь при дефолте, EAD — сумма задолженности. Модель позволяет отказывать клиентам с высоким PD, тем самым уменьшая общую сумму потерь.

**Доля одобренных заявок при фиксированном уровне риска**
Показывает, сколько заявок банк может одобрить, удерживая долю дефолтов на заданном уровне (например, не выше 5%). Хорошая модель позволяет одобрить больше клиентов без роста риска, увеличивая кредитный портфель и прибыль.

## A/B-тестирование
Для сравнения текущей модели (v1) с новой версией (v2) предусмотрен A/B-тест. Трафик разделяется случайным образом 50/50, длительность определяется необходимым объемом выборки для статистической значимости. 

Основная метрика - F1 score класса дефолта, дополнительная - Precision. Статистический анализ проводится с помощью Z-теста для пропорций и построения доверительных интервалов. 
Подробный план описан в ab_test_plan.md.
